Bridge 연결 확인

1. Bridge 연결

대기

WebView에서 addJavascriptInterface로 등록한 window.Android 객체를 확인합니다.

1. Holder 지갑 관리

미생성

여러 개의 지갑을 생성하거나 복구하여 관리할 수 있습니다.

복구는 seed 또는 12단어 비밀구문을 지원합니다. 복구 시 holder auth key가 새로 생성되므로 자동 재등록을 끄면 복구 성공 후 Holder DIDSet 등록을 직접 다시 실행해야 verifier 제출이 통과합니다.

1-1. 자산 조회

대기

입금 확인과 송금 전 검증을 위해 현재 XRPL 계정의 XRP 잔액과 trust line 자산을 조회합니다.

1-2. 입금

대기

입금용 주소 표시, 복사, QR 미리보기를 제공합니다. 실제 입금 확인은 자산 조회로 다시 확인합니다.

DEPOSIT QR

1-3. XRP 송금

대기

활성 인증 세션과 송금 직전 네이티브 재인증이 필요합니다. XRP 소수 입력을 우선 지원하고, 필요하면 drops 값을 직접 넣을 수 있습니다.

1-4. 거래내역

대기

최근 XRPL account_tx 결과를 확인합니다. 1차 구현은 최근 내역과 Payment 중심 방향만 표시합니다.

2. 실서버 issuer / verifier

대기

신규 기본 흐름은 dc+sd-jwt credential과 SD-JWT+KB presentation입니다. Android holder 앱은 issuer secret/PEM을 다루지 않습니다.

발급 응답의 credential 원문은 전체 disclosure를 포함하므로 로그 공유 시 주의하세요.
개발용 도구 A: XRPL 발급 디버그 (EXPERIMENTAL)

XRPL 발급(디버그)

EXPERIMENTAL

개발/디버그 전용 카드입니다. 운영 흐름에서는 사용하지 않습니다. Issuer가 먼저 CredentialCreate로 ledger entry를 만듭니다. subject 계정은 XRPL testnet에서 이미 활성화되어 있어야 합니다.

순서: 발급 → 저장 → 수락 → 상태 조회
발급만 먼저 끝내고, holder 수락은 아래 별도 카드에서 진행합니다.

3. VC 저장 / 상태 확인

미저장

발급된 SD-JWT 원문을 저장하고 XRPL accepted 상태를 확인합니다. 실제 검증은 위 실서버 verifier 버튼을 사용합니다.

Corporate KYC Credential
테스트 법인 주식회사
123-45-67890APPROVED

4. CredentialAccept 제출

대기

issuer가 만든 ledger entry를 holder가 수락합니다. 이 카드만 눌러도 되도록 필요한 입력을 모았습니다.

holder account가 testnet에서 활성화되어 있지 않으면 `tecNO_TARGET`가 뜹니다.

5. Key 서명 / VP

대기

Verifier nonce/aud를 받은 뒤 selected SD-JWT에 KB-JWT를 붙여 제출합니다.
